# 口播稿规范 — 老师怎么讲

`口播稿.md` 只回答一个问题：这条领读视频里，作者 / 主讲人要怎么对「你」讲，才顺、才自然、才讲得透。

它不写模板、不写秒数、不写画面指令、不写 reveal 标记。画面节奏属于 `分镜.js` 的 atoms / steps；口播稿只保留可被 TTS 念出来的话。

## 一、备读内容

阅读单元备读内容：

```text
books/<书>/chNN/NN.md
```


## 二、声音身份

- **作者 / 主讲人第一人称**：道德经就是老子，论语就是孔子，理想国多用苏格拉底式领读。不要写成第三方旁白。
- **只对「你」说**：像老师当面带读。少说「大家」「读者」「我们」。
- **白话、密、不断水**：不堆学术术语，不照搬注家目录。把注家的差异消化成听得懂的话。
- **以讲懂原文为主线**：生活关联只服务于理解原文，不把视频变成人生鸡汤。

## 三、结构

一条口播通常按四段走：

1. **开场**：报书名章号 + 一个钩子。
2. **引子**：接上前文，说明这一章 / 单元要看什么。
3. **逐句精讲**：顺着原文讲，重点深讲，非重点带过。
4. **收束**：用一句判断、反问或点睛，停在这一段的劲上。

结构可以有小标题，但 TTS 只读取 `>` 引用块里的内容。

## 四、讲法

逐句精讲时，通常按这个节奏：

```text
引出原文
→ 用白话讲透字面
→ 点关键字 / 关键关系
→ 落到经验里看
→ 收出这一句的劲
```

不要每句机械凑满五步。真正要保证的是：听完之后，用户能知道这句原文为什么这么说、它在整章里承担什么作用。

## 五、读音校对

TTS 对文言多音字、生僻字容易读错。agent 送配音前，在正文前列读音校对表。

做法：

```markdown
**读音校对表（送 TTS 前处理）**

- 处：chǔ，表示安处、居于。
- 恶：wù，表示厌恶。
```

正文中需要时写成：

```text
处（chǔ）
```

`gen-tts.js` 会剥掉括号注音，只把字送去配音；字幕仍能保留注音帮助观看。

常见坑：

- 处 chǔ
- 恶 wù
- 几 jī
- 乐 yuè
- 王 wàng（动词）

拿不准就标。不要为了省事让 TTS 猜。

## 六、parse 契约

`gen-tts.js`、`draft-storyboard.js`、`align-durs.js` 共用同一个解析规则：

- 首个 `---` 之前：标题、voice 说明、读音校对表，不进配音。
- 首个 `---` 之后：每组连续 `>` 引用行 = 一条旁白段。
- 空行、`##` 标题、`---` 分隔线会断段。
- 每段旁白必须以 `。！？!?` 结尾。
- `>` 行只放要讲出口的话。

口播稿里不要写：

- 画面说明。
- 模板名。
- 秒数。
- `[[marker]]` 一类视觉锚点。
- 给分镜作者看的备忘。

## 七、文件骨架

````markdown
# 《书名》第 N 章「章题」· 领读口播稿

> voice = 作者第一人称领读老师，对「你」说；白话、密、不学术腔。

**读音校对表（送 TTS 前处理）**

- 〈多音字 / 生僻字〉：〈读音与理由〉

---

## 〇 · 开场
> 今天，我们读《书名》第 N 章。〈一句钩子。〉

## 一 · 引子
> 〈接上前文，说明这次要看什么。〉

## 二 · 逐句精讲
> 〈第一句原文的讲解。〉
> 〈下一句原文的讲解。〉

## 三 · 收束
> 〈一句判断 / 反问 / 点睛，停住。〉
````

## 八、写完后的自检

- 从第一个 `>` 连续读到最后一个 `>`，是不是一条自然的领读？
- 有没有为了画面而写进去的非口播内容？
- 每段是否都有句末标点？
- 多音字是否已校对？
- 讲法是否围着原文，而不是绕开原文做感想？
