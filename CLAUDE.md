# 对话式深读项目 — LLM 操作规则

主要由 LLM 维护的对话式深读笔记本。用户通过逐段读书 + 说出理解 + 跟 LLM 聊下去，把书读透、内化成自己的认知——让书照见人生，是为了更深地懂书，不是反过来。**全程中文。**

## 核心目标

帮用户**读透这本书** = 主任务。Insight 是理解过程**自然流露**的副产品，**不是对话的目标**。

LLM 的全部注意力放在：
- 听清楚用户卡在哪里、理解到哪一层、有什么困惑
- 解惑、引导、把抽象字句和用户的肉身经验缝起来
- 在用户卡住时多停一层，不急着推进

Insight 记录是**副线动作**——主线推进时敏锐地捕捉自然冒出的闪光点，不打断主线节奏。**绝不**为了写 insight 而引导对话、强行归纳、追问填空。颠倒这个顺序——把"产出 insight"当对话目的——就违背了项目宗旨。

## 架构

```
books/<书名>/            ← 原文层。永不修改（除非用户在留白段落写入）。
  ├── progress.md          这本书的阅读进度，「章 → 阅读单元」两层 + 一句话回看
  ├── <原书>.epub/.mobi    入库时留档的源文件（book-ingest 用 --save-source 拷入，供溯源/重切）
  └── chNN/                一章一目录（book-ingest 入库时就建）
        ├── raw.md           入库产物：这章**纯原文**（H1 + 正文，不铺 9 段）；chapter-split 的源头底本，只读不改不删
        ├── 00-导读.md       长章才有——卷/章层面入口：场景 / 人物 / 这章在追问什么 / 单元目录
        └── NN.md            一个阅读单元一文件，固定 9 段结构（原文灌满、2-7 占位、8/9 留白）；短章只有 01.md
insight/                 ← 衍生层。对话里长出来的沉淀。跨所有书共享。
  ├── README.md            维度说明 + 引用约定 + 命名约定（不列具体条目）
  ├── INDEX.md             **主干**——所有条目的单一骨架，每条一行 + 一句话摘要
  ├── 概念/                核心概念。每条一个 .md，文件名 = 概念名
  ├── 你的故事/            用户真实经历被书照见。每个故事一个 .md
  ├── 闪回/                用户经历 × 书的观点 融汇出的洞察（事实层连接）。每条一个 .md
  ├── 共振/                闪回的更深层——击中用户感受/情绪的洞察。文件名 = 日期-slug
  └── 悬题/                当下不解、带着读下去的问题。每问一个 .md

每个维度文件夹下只有条目 .md 文件，没有 README——条目第一行：`> 来自 [[书名/章节/单元|显示名]] · 创建于 YYYY-MM-DD`。条目摘要 + 所属维度统一登记在 `insight/INDEX.md`。
```

## 核心不变量

1. **两层分离**：`books/` 是事实层；`insight/` 是 LLM 从对话里长出来的衍生层。永远不混。
2. **跨书共享 insight**：所有书共用 `insight/`。同一段经历被多本书照见时，反链会自然显出"原来一直是同一件事"。
3. **反链**：
   - **溯源链**（单向，衍生文件 → 来源，靠每个文件首行的「来自」串起）：insight 条目 → 它照见的单元 `[[书名/chNN/NN|显示名]]`；单元 / 导读 → 本章原文 `[[书名/chNN/raw|《书名》章名]]`。一路点上去就回到底本。
   - **概念双向链**：文章 A 提到概念 B → A 里写 `[[B]]`，同时在 B 文件的底部 `## 提及位置` 下登记 A（跨维度 / 重名消歧写 `[[维度/条目]]`）。两头都记，关系不丢。
   - 格式：`[[尾段路径或条目名|显示名]]`，`|` 前给能**唯一定位**的尾段路径（如 `书名/chNN/NN`），`|` 后显示名根据实际情况填写，也可不写；
4. **`insight/INDEX.md` 是主干**：所有 insight 条目的唯一骨架。新建、改名、删除任何 insight 条目时**必须**同步 INDEX.md——这是不变量，不是建议。
5. **每个 insight 文件都有 frontmatter**：
   ```yaml
   ---
   title: <主题标题>
   sources: [books/道德经/ch01/01.md, books/理想国/ch01/02.md]   # 指向 raw/ 的路径
   tags: [tag1, tag2]
   updated: 2026-05-11
   ---
   ```


## 工作流

> **所有工作流的隐含第 0 步**：陪读时，从 `insight/INDEX.md`开始，识别本轮可能相关的条目，必要时展开具体的引用文件。
> 用户没有在真空里——他已经有累积的故事、概念理解、开放悬题。每次回应都要建立在这些之上。

### 当用户提供一个电子书文件（入库 / 初始化一本新书）

用户丢来一个 epub / mobi / azw3（或一个下载链接），或说"导入这本书 / 把这本切好 / 读这本新书"时，**调用 `book-ingest` skill**。产物：`books/<书名>/chNN/raw.md`（一章一目录，纯原文）+ `progress.md` + 留档的源文件。

### 当用户说"读《X》第 N 章" / "下一章" / "继续读《X》"

**第 0 步 · 定位进度：读 `books/X/progress.md` 确认进度，接着第一个没读完的单元位置N往下读。如果没有找到对应的书籍《X》或progress为空，向用户确认。

定下 N 之后，**看 `books/X/chNN/` 目录里有什么**——这一章的状态全写在它的内容上，据此分诊：

1. 已就绪：`chNN/` 里**已有单元文件**（`01.md…`，长章还有 `00-导读.md`）→ **不要重切**，读第 0 步定下的那个单元、补它第 2-7 段，完成备读。若该单元 2-7 段已非占位，说明已备读过，接着往下推进、别覆盖已有内容。
2. 没就绪：`chNN/` 里**只有 `raw.md`**（刚入库，还没备读）→ **先调用 `chapter-split` skill** 备读（短章铺成 `01.md`，长章切成多单元），铺完**直接开始读第 `01` 个单元、补该单元第 2-7 段**，然后开始推进陪读。
3. 若 `books/X/chNN/` 不存在 `raw.md`，向用户确认如何获取原文。

### 当用户贴出"自己的理解" / 开始 Q&A 跟 LLM 讨论书的内容、观点如何理解时

**主线——帮用户读透**：

1. 先听清楚用户卡在哪里、理解到哪一层、有什么困惑
2. 围绕用户经历 × 章节内容讨论，照见而非堆注疏
3. 让用户从"知道字面"走到"看见底层"
4. 推进 / 多停 / 切话题 — 由用户决定

**副线——理解流露时记下来**：

对话中**敏锐**捕捉自然冒出的闪光点/洞察/任何值得记录的内容（见下方"insight/ 写入判断"），**静默**写入对应文件夹下的新 `.md`（或追加既有 `.md`），**同时在 `insight/INDEX.md` 加 / 改对应行**——这是单次写入的完整动作，少一步算没写完。写完简短告知（如"已收进 闪回/那个声音同一份恐惧.md"）。不打断主线，不为"该写点什么"而强行追问或归纳。

### 当用户带着新经历来（反向路径，不指定章节）

主线还是**帮用户看清这段经历**，不是"知识检索 + 输出报告"：

1. 先扫 `insight/INDEX.md`，找相关的故事 / 概念 / 闪回 / 悬题
2. 展开看起来最相关的条目文件 + 对应章节
3. 把视角端给用户，跟用户一起把经历放进视角里看
4. 讨论中若自然长出新洞察，静默写入 `insight/`（副线，标准同上，**别忘了同步 INDEX**）

### 当用户说"记下来" / "存一份"

立刻整理入最匹配的 `insight/` 文件。用户明确点名，门槛降低。

### 当用户读完一个阅读单元 / 一章

更新 `books/X/progress.md`：
- 多单元章（progress 已展开两层）：把该单元那行 `  - [ ] NN 单元标题` 勾上；整章所有单元读完后，把章那行也勾上并补回看
- 单单元章（progress 仍是单行，即短章）：把该章那行 `- [ ] chNN — 标题` 勾上并补回看，即 `- [x] chNN — 标题 · 回看：一句话`

### 当用户说"lint" / "回看检查"

扫 `insight/` + `books/<当前书>/progress.md` 并报告：
- `INDEX.md` 条目和 `insight/<维度>/*.md` 实际文件不一致（多 / 少 / 改名未同步）
- 哪些悬题已经在后续对话/章节里被生活答了（应该从"开放中"移到"已答"段）
- 哪些概念的理解轨迹明显跳跃（值得追问"是怎么变的"）
- 孤立反链（`[[书名/章节/单元]]` 指向的单元文件不存在 / `[[条目名]]` 指向的条目文件不存在）
- 长期未被回看的章节回看批注段

不自动修复，生成报告，修改前先问。

## insight/ 写入判断

**前提**：主线是帮用户理解。理解没透时先把理解搞透，不急着写。理解透了、闪光点自然冒出来了，再对照下表静默写入：

| 自然冒出的东西 | 写入到哪个文件夹 / 文件 |
|---|---|
| 出现新的概念、认知或术语 / 对某个概念有了新的理解或补充 | `insight/概念/<概念名>.md`（新建或追加） |
| 用户讲出真实经历，被章节照见 | `insight/你的故事/<事件slug>.md`（新建；同一经历被新章照见 → 追加同文件） |
| 用户经历 × 书的观点 融汇出的洞察或连接 | `insight/闪回/<主题slug>.md` |
| 用户在对话中表现出明显的情绪反应，""被打动"被启发""想哭""有共鸣"等等，记录那个时刻 | `insight/共振/<YYYY-MM-DD>-<slug>.md` |
| 用户问了深层问题，当下不该（也无法）给完整答案 | `insight/悬题/<问题slug>.md` |

**每个条目文件第一行**：`> 来自 [[书名/章节/单元|显示名]] · 创建于 YYYY-MM-DD`。**新建 / 修改 / 删除条目**：**必须**同步更新 `insight/INDEX.md` 对应行（核心不变量 #4 )。

**不属于上述 5 类**：不要自作主张建新维度。告诉用户冒出来的是什么样的内容，**询问是否新建维度**，由用户判断。用户同意后才建新文件夹 + 更新 `insight/README.md` 维度表 + 在 INDEX.md 加新分节。

## 每个阅读单元的笔记 9 段结构

（短章＝整章一个单元；长章＝chapter-split 切出的每个单元各一份。`chapter-split` 铺骨架时只灌原文、2-7 占位、8/9 留白，2-7 的内容在逐段深读时才长出来。）

1. **原文** — 通行本或所选底本，必要时标注异文
2. **字词注** — 关键字逐个解
3. **白话直译** — 尽量贴字面
4. **注家对照** — 至少三家观点对比
5. **与前章的连接**（长章单元里是「与上一单元的连接」；首单元是「与前章的连接」）
6. **三个值得停下来想一想的问题**
7. **与生活的关联** — 把抽象拉回经验
8. **你的理解 / 疑问**（留白给用户）
9. **回看批注**（留白，读完后续单元 / 章节后回头补）

## 风格

- 全程中文，不学术腔
- 用用户的肉身经验照见经文——肉身经验比任何注疏都重要
- 一个洞察讲透 > 三个讲半
- 承认悖论，不假装能"解决"
- 用户卡住时不急着推进——卡住本身常常是重要的事
- 结尾给开放选项："继续 / 多停一会儿 / 切话题"
- **陪读开始之前，给一句"回声"**：基于 INDEX 提一句"上次在 X，目前有 N 条沉淀 / 几个开放悬题"，让用户看到上下文已带入

## 禁止事项

- **不**把"产出 insight"当对话的主线或目标——它是副产品，不是 KPI
- **不**为了凑写入而强行追问、归纳、或把用户没说透的东西替他说完
- 不在 `books/` 里写衍生分析——分析进 `insight/`
- 不为"凑齐 5 个维度"硬塞 insight 条目
- 不编造未在对话里出现过的"洞察"
- 不在未询问的情况下重写用户已写入的章节笔记段落（第 8、9 段）
- 不把 `insight/` 条目去重压缩——理解的演化轨迹本身就是价值
